# Debugger Framework · 多平台适配说明

本文档说明如何在 5 个支持的平台上部署与使用 Debugger Framework，并解释各平台的适配层目录结构、frontmatter 规范、Hook/Skill 的支持差异，以及如何从 SSOT 同步更新。

> 路径约定：除非特别说明，以下路径均相对于 `extensions/debugger/`。

---

## 目录结构总览

```
extensions/debugger/
├── .claude/
│   └── settings.json                 # Claude Code 原生 Hooks（可阻断）
├── README.md
├── common/                           # 平台无关 SSOT（Single Source of Truth）
│   ├── AGENT_CORE.md                 # 9 个 Agent 的权威定义 + 合同约束
│   ├── agents/                       # 9 个 Agent Prompt（SSOT）
│   ├── knowledge/                    # spec/library/traces/templates
│   ├── hooks/                        # 质量 Hook：validators + schemas + utils
│   └── project_plugin/               # 项目级 Plugin 规范与示例
├── design/                           # 早期草案（以 common/ 为准）
├── docs/                             # 说明文档（本文件）
├── platforms/                        # 各平台适配层（由脚本同步生成）
│   ├── claude-code/
│   ├── code-buddy/
│   ├── claude-work/
│   ├── copilot/
│   └── manus/
└── scripts/
    ├── sync_platform_agents.py       # 从 common/agents 同步到 platforms/*
    └── validate_tool_contract.py     # 校验 prompt/trace 中 rd.* 工具名与参数
```

**核心原则：**
- 修改 Agent 逻辑 → 只改 `common/agents/`（SSOT）
- 同步到平台适配层 → 运行 `python3 scripts/sync_platform_agents.py`
- 工具名/参数的唯一真值 → `extensions/rdx-mcp/rdx/spec/tool_catalog_196.json`

## 平台一：Claude Code

### 文件结构
```
platforms/claude-code/agents/
  01_team_lead.md
  ...
  09_report_knowledge_curator.md
```

### frontmatter 格式（示例）
```yaml
---
name: "Debugger Team Lead / Orchestrator"
description: "Coordinate delegates and enforce quality gates"
agent_id: "team_lead"
model: "claude-sonnet-4-5"
tools: "bash,read"
color: "#E74C3C"
---
```

### 质量 Hooks（强制）
Claude Code 使用 `.claude/settings.json` 配置原生 Hooks：
- **PostToolUse**：BugCard 写入 `common/knowledge/library/bugcards/` 时做 schema/合同校验
- **PostToolUse**：Skeptic signoff 写入 `common/knowledge/library/sessions/**/skeptic_signoff.yaml` 时做格式校验
- **Stop**：仅在输出“结案/最终裁决”标记时触发最终门禁（反事实 + Skeptic gate + artifacts 完整性）

## 平台二：Code Buddy

### 文件结构
```
platforms/code-buddy/
├── .codebuddy-plugin/plugin.json     # 插件清单
├── .mcp.json                         # rdx-mcp 服务器配置
├── agents/                           # 9 个 Agent（脚本同步生成）
├── hooks/hooks.json                  # 原生 Hook 配置（PostToolUse + Stop）
└── skills/debugger/SKILL.md        # 调试知识 Skill（动态索引 spec）
```

### 质量 Hooks（强制）
`platforms/code-buddy/hooks/hooks.json` 通过命令钩子运行：
- `common/hooks/utils/codebuddy_hook_dispatch.py write-bugcard`
- `common/hooks/utils/codebuddy_hook_dispatch.py write-skeptic`
- `common/hooks/utils/codebuddy_hook_dispatch.py stop-gate`

## 平台三：Claude Work

### 文件结构
```
platforms/claude-work/
├── plugin.json                       # agents + knowledge_base + mcp_servers
└── agents/*.md                       # 9 个 Agent（脚本同步生成，文件名不同于序号版本）
```

### 注意
- Claude Work 平台侧没有强制的系统级 Hook（依赖各 Agent prompt 内嵌的“质量门槛检查清单”）
- `plugin.json` 中 `agents[].name` 仅用于 UI 展示；路由/审计必须使用 `agent_id`（见 `common/AGENT_CORE.md`）

### 文件结构
```

```

### 注意
- 以工作流步骤描述替代多 Agent prompt
- 质量门禁以工作流关卡形式体现（不同于 Claude Code/Code Buddy 的系统级阻断）

## 跨平台一致性校验与同步

### 1) 工具合同校验（推荐在每次改 prompt 后运行）
```bash
python3 scripts/validate_tool_contract.py --strict
```

### 2) 同步平台 prompts（每次改 common/agents 后必须运行）
```bash
python3 scripts/sync_platform_agents.py
```
