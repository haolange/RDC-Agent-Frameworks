# Debugger 平台模板维护说明

本文档只适用于 `debugger/` 的平台模板体系，用来说明模板目录 contract、`common/` 覆盖工作流、scaffold 生成与校验规则。

它是 `debugger` 维护者文档，不是仓库级规则，也不是运行时共享文档。

平台 enforcement tier、guard 注入点和 public entrypoint 真相统一由 [`platform-enforcement-matrix.md`](./platform-enforcement-matrix.md) 定义；本文不再单独发明第二套 hooks / pseudo-hooks / audit-only 口径。

## 目录职责

- `debugger/common/`：`debugger` 唯一共享真相，用户需要把它拷贝到目标平台模板根目录的 `common/`。
- `debugger/common/docs/`：`debugger` 唯一运行时共享文档入口，平台薄包装只允许引用这里。
- `debugger/docs/`：`debugger` 维护者文档区，不给平台 runtime 直接依赖。
- `debugger/platforms/<platform>/`：`debugger` 的宿主薄包装模板，只保留宿主必须识别的入口文件、少量平台专属配置，以及等待 `debugger/common/` 与 `tools/` 覆盖的最小占位目录。
- `debugger/platforms/codex_plugin/`：Codex installable plugin 的外层包装目录；真正的 plugin root 固定为其中的 `rdc-debugger/`，外层只放安装说明与 marketplace 示例。

## 用户工作流

1. 选择一个 `platforms/<platform>/` 模板。
2. 若选择 `codex_plugin`，先进入其内层 plugin root `platforms/codex_plugin/rdc-debugger/`；外层 `platforms/codex_plugin/` 不参与运行时接线。
3. 将根目录 `debugger/common/` 整体拷贝到该模板根目录的 `common/`。
4. 将 `RDC-Agent-Tools` 根目录整包拷贝到该模板根目录的 `tools/`。
5. 运行 `python common/config/validate_binding.py --strict`，确认 package-local `tools/`、required paths、zero-install runtime、snapshot 与宿主入口文件全部对齐。
6. 使用该模板根目录下、与 `common/` 和 `tools/` 并列的预生成 `workspace/` 作为运行区。
7. 完成覆盖后，再在对应宿主中打开模板根目录使用。

若当前模板是 `codex_plugin`，完成第 1-7 步后还需要：

- 将内层 `rdc-debugger/` 同步到 `~/.agents/plugins/rdc-debugger/`。
- 将外层 `references/personal-marketplace.sample.json` 合并到 `~/.agents/plugins/marketplace.json`。
- 在 Codex 的 `/plugins` 中安装或刷新 `rdc-debugger`。

未完成 `common/` 覆盖、`tools/` 覆盖与 binding 校验前，平台模板不可用。

## 维护规则

- 共享运行时文档只维护在 `common/docs/`。
- 平台薄包装中的 agent、skill、README、references 只做宿主壳与引导，不复制共享正文。
- `common/README.md` 是 shared common 覆盖后的正式入口说明，不再承担平台占位说明职责。
- 平台模板中的 `common/` 默认只保留单文件占位；不再为未覆盖状态生成伪完整 placeholder 文件集。
- 平台模板中的 `tools/` 默认只保留单文件占位；正式工具真相只来自复制后的 package-local `tools/`。
- `platform_adapter.json` 是共享 adapter manifest；`paths.tools_source_root` 固定为 `tools`，且 live runtime 固定为 `worker_staged`，不再支持手工配置。
- 平台 `MCP` opt-in sample 与共享 `mcp_servers.json` 必须统一包装 `tools/rdx.bat --non-interactive mcp`，不再以系统 `Python` 为正式前提。
- 平台模板必须继续预生成 platform-local `workspace/` 占位骨架，保证平台根目录下始终存在与 `common/`、`tools/` 并列的 `workspace/`；仓库根目录不维护共享 `workspace/`。
- 若 `debugger/` 新结构替代旧结构，旧目录、旧脚本名、旧双轨引用必须在同一轮删除。
- Claude Code 的 `hooks.matcher` 只使用 string；文件路径过滤由共享 hook dispatcher 读取 hook payload 后完成，不在 settings 里写 object matcher。

## Scaffold 与校验

生成平台模板：`python debugger/scripts/sync_platform_scaffolds.py`

纯检查模式：`python debugger/scripts/sync_platform_scaffolds.py --check`

布局校验：`python debugger/scripts/validate_platform_layout.py --strict`

工具契约校验：`python debugger/scripts/validate_tool_contract.py --strict`

手工接线校验：`python common/config/validate_binding.py --strict`

## 平台路径对齐原则

- Claude Code 使用 `.claude/settings.json`、`.claude/CLAUDE.md`、`.claude/agents/*.md`。
- Codex workspace-native 模板使用 `AGENTS.md`、`.agents/skills/*/SKILL.md`、`.codex/config.toml` 与 `.codex/agents/*.toml`。
- Codex installable plugin 使用 `.codex-plugin/plugin.json`、`skills/*/SKILL.md`、`references/mcp-opt-in.sample.toml` 与外层 `references/personal-marketplace.sample.json`。
- GitHub Copilot CLI 使用 plugin 目录形态与 `agents/*.agent.md`、`skills/*/SKILL.md`。
- GitHub / VS Code Copilot 使用 `.github/agents/*.md`、`.github/skills/*/SKILL.md`、`.github/mcp.json`。
- Code Buddy、Claude Work、Manus 保留各自宿主要求的 manifest、workflow、references 文件，但共享正文仍以 `common/` 为准。
